# 第四卷总结

第四卷做的事情，不是再补更多知识点，而是把前面所有知识重新压缩成几条稳定骨架。

## 1. Claude Code 不是单层系统

它至少同时拥有：
- 入口层
- 控制面层
- 执行制度层
- 扩展协作层
- 接口与传输层

这意味着理解 Claude Code 不能只盯住某一个熟悉模块。只看 CLI，会低估 QueryEngine；只看 Tool，会低估 Permission 与 Hook；只看 Agent 或 MCP，又会低估入口、上下文与 transcript 这类底层秩序。

## 2. 一次请求会穿过整套系统

用户输入不会直达模型，也不会在模型处结束；它会持续穿过上下文装配、执行管线、权限制度、扩展接口与 transcript 回写。

所以 Claude Code 真正提供的不是“回答”，而是一条被运行时持续治理的会话链。这个判断也是第四卷最核心的收束点：会话连续性，比单点能力更接近系统本质。

## 3. 真正值得抓住的是设计方法

Claude Code 反复展示出几种稳定方法：
- 尽早分流
- 制度化边界
- 多层状态治理
- 扩展与协作的协议化接入

这些方法之所以重要，是因为它们不是一次性技巧，而是在入口、控制面、执行面、扩展面、接口层被重复应用的结构原则。

## 4. 为什么“源码阅读地图”比“模块清单”更重要

第四卷最后真正留下来的，不该只是“我知道有哪些模块”，而应该是：

- 我知道先从哪一层读起；
- 我知道某个模块在整套系统里的位置；
- 我知道一个功能点背后往往连着哪些制度边界与状态层；
- 我知道该怎样从站点笔记重新回跳到整体系图。

这也是本书把原始 `note/` 语料重组成 `notebook/` 书稿的意义：不是丢掉原站点，而是让原站点第一次能被放进稳定骨架里阅读。

## 5. 全书最终判断

因此全书最终留下的判断是：

> Claude Code 最有价值的，不只是它“支持很多功能”，而是它把一个不断扩张的 CLI 智能体系统，组织成了依然可以被理解、被治理、被扩展的复合架构。

这也意味着，整本书最终要交给读者的，不只是一些模块知识，而是一张能够反复回用的系统地图：以后无论再回读某个站点、某个模块，还是某条执行链，都能更快判断它在整套系统里究竟处于哪一层、承担什么角色。
