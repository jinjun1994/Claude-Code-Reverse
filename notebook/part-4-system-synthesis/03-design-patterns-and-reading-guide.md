# 第 13 章：设计模式、覆盖矩阵与阅读地图

前面几章已经给出了系统结构与生命周期。本章最后做三件事：

1. 提炼全书最关键的设计模式；
2. 给出模块覆盖矩阵；
3. 提供后续跳读的地图。

## 13.1 三类最关键的设计模式

### 一，尽早分流
出现在：CLI 入口、工具池装配、启动路径。

核心价值：
- 不命中的路径不支付成本；
- 特殊模式不拖累主路径；
- 大系统仍然可以维持首响性能。

### 二，制度化边界
出现在：Tool 接口、Permission Runtime、Hooks、MCP 配置策略。

核心价值：
- 能力先被定义，后被调用；
- 开放执行能力仍可验证；
- 系统不会因为扩展而失去秩序。

### 三，多层状态与上下文治理
出现在：Bootstrap State、App State、QueryEngine、Compact、Memory、Transport。

核心价值：
- 会话能持续；
- 对话可压缩、可恢复；
- 多智能体与远程形态不会直接打碎主会话一致性。

## 13.2 再向下细看：这些模式如何互相咬合

如果把全书放在一起看，这三类模式并不是平行并列，而是层层支撑：

- **尽早分流** 负责控制复杂度入口，不让所有路径把成本拖到同一层；
- **制度化边界** 负责控制能力扩张，不让开放执行面变成失控执行面；
- **多层状态治理** 负责控制长期连续性，不让会话、agent、remote、memory 彼此打架。

也就是说，Claude Code 的系统感，恰恰来自这三种模式被重复应用在不同层面。

## 13.5 这一章真正提供的阅读姿势

最后这一章想留下的不是更多细节，而是一种阅读姿势：

- 不再把每一站都当作孤立观察点；
- 也不把 Claude Code 误认为一套平铺模块；
- 而是把它看成一套会沿着“入口 → 控制面 → 扩展面 → 接口层”不断递进的复合系统。

一旦抓住这几条骨架，再回头读任何站点，都会更容易知道它究竟在系统里扮演什么角色、为什么被设计成这样。

## 13.6 本章小结

> Claude Code 最适合被理解成一套从入口、控制面、扩展面到接口层层递进的复合系统。只要抓住这几条骨架，再回头读任何站点，都会更容易知道它究竟在系统里扮演什么角色。
