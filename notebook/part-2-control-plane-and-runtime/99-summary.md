# 第二卷总结

第二卷真正回答的是：Claude Code 怎样把一次会话做成一台稳定机器。

本卷可以压缩成四条关键认识：

## 1. 配置层决定运行边界

Config / Auth / Settings 不是外围便利功能，而是在决定：

- 谁是可信来源；
- 哪些策略能覆盖哪些行为；
- 这次会话究竟以什么权限与模式运行。

## 2. QueryEngine 让 Claude Code 成为“会话系统”

真正的主循环不是单纯聊天，而是：

- 组装上下文；
- 调模型；
- 分流工具；
- 回写 transcript；
- 处理压缩、恢复与预算。

## 3. Tool 系统把能力变成制度化对象

Claude Code 的强大不只是因为它有 Bash、Read、Grep、Agent、MCP，而是因为所有这些能力都被统一装进了同一套接口、权限与结果回写机制里。

## 4. Permission、Hook、Session Runtime 共同构成控制面

它们决定的不是“有没有某个功能”，而是：

- 什么能发生；
- 谁可以决定它发生；
- 发生之后系统如何继续保持稳定。

## 5. 为什么第二卷是全书的执行中枢

第一卷建立了入口与会话外壳，第三卷展开了扩展与协作形态，但真正把这些能力持续组织起来的，仍然是第二卷所描述的控制面。

也正因为如此，第二卷既是全书最像“运行时内核”的部分，也是最能体现 Claude Code 系统感的部分：它把模型输出、工具副作用、权限裁决、Hook 插入、状态恢复，全部重新编排成一条单一会话时钟。

所以第二卷最终留下的判断是：

> Claude Code 的控制面与运行时，不是在支撑一个模型回答问题，而是在支撑一台长期运行、可以主动行动、还能被制度化约束的会话引擎。

## 与第三卷的关系

一旦第二卷回答了“系统怎样稳定运行”，第三卷才会进一步回答：

- 外部能力怎样接进来；
- 多智能体怎样被组织起来；
- 本地会话怎样被延展到远程与协议化环境。

换句话说，第二卷解决的是运行秩序，第三卷解决的是能力边界。
