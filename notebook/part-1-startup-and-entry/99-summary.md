# 第一卷总结

第一卷处理的是 Claude Code 最容易被低估的一层：启动。

很多人会自然地把它理解为“一个命令行程序，起起来以后进入主循环”。但从源码角度看，这种理解远远不够。

本卷真正建立的是三条认识：

## 1. 入口是一种宿主选择系统

`entrypoints/cli.tsx` 的意义不在于“接收参数”，而在于尽早判断：

- 这是快速路径还是完整 CLI？
- 这是本地交互会话还是特殊宿主？
- 哪些模块根本不该在当前路径加载？

## 2. 初始化是一种顺序管理系统

`main.tsx`、`entrypoints/init.ts`、`setup.ts` 共同说明，Claude Code 的启动不是堆功能，而是在管理顺序：

- 什么必须先完成；
- 什么可以后台继续；
- 什么会影响后续所有工具/子进程/会话；
- 什么必须在进入 REPL 之前就变得稳定。

## 3. 启动的终点不是函数返回，而是会话成立

命令系统、Bootstrap State、REPL、Ink 组件与终端渲染一起说明：

- 会话必须有身份；
- 状态必须可订阅；
- 输出必须可见；
- 交互必须可持续。

## 4. 为什么第一卷虽然最靠前，却不只是“铺垫卷”

如果把第一卷只看成背景知识，就会漏掉它在全书里的真正地位。入口与启动阶段并不是外围杂务，而是在最早时刻就决定：

- 哪条运行路径会被真正拉起；
- 哪些模块根本不会进入本次会话；
- 后面控制面、扩展面、接口层要站在什么宿主条件之上运行。

也就是说，第一卷不只是“先讲一点前置”，而是在最开头就决定整套系统接下来要活成什么形态。

## 5. 为什么“尽早分流”会从这里一路贯穿到全书结尾

第一卷最先暴露出的核心设计模式，就是尽早分流：未命中的路径不支付成本，特殊模式不拖累主路径，复杂系统在入口处先做宿主判断。

这一模式后面会一再重现：

- 第二卷里，QueryEngine 会把文本收束、tool_use、权限裁决、回写过程尽早分成不同分支；
- 第三卷里，MCP、Agent、Remote 会先以协议化或制度化方式进入系统，而不是直接硬接；
- 第四卷最后，又会把这条原则重新提炼成整套系统的稳定设计方法。

所以第一卷不是孤立的启动说明，而是整本书设计语言的第一次显影。

所以第一卷最终留下的判断是：

> Claude Code 的启动层，本质上是在回答“怎样把一堆可能的入口，变成一个稳定可交互的运行世界”。

## 与下一卷的关系

一旦这个世界成立，第二卷才真正开始：

- 配置如何合并？
- 权限如何裁决？
- Tool 如何装配？
- QueryEngine 如何把一轮对话跑完？

也就是说，第一卷搭的是舞台，第二卷才开始讲表演如何发生。
