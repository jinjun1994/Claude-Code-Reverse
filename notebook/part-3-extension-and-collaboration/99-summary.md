# 第三卷总结

第三卷真正回答的是：Claude Code 怎样超出“本地单会话”这件事。

## 1. 扩展性来自正式接入口，而不是特例堆积

MCP、Plugin、Skills 共同说明：系统必须先决定怎样接纳外部能力，然后才谈这些能力具体做什么。

## 2. Memory 让会话拥有时间维度

有了 Memory，系统不再只依赖当前上下文窗口，而能把长期信息纳入后续运行。

## 3. AgentTool 与 Swarm 让系统拥有组织维度

有了 agent、mailbox、team file、task list、permission sync，Claude Code 才真正从“单体会话”进入“多行动者协作系统”。

## 4. Bridge、Remote 与 Transport 让系统拥有空间维度

本地与远程、交互与协议、UI 与执行环境被重新缝合，Claude Code 才能跨机器延展而不失序。

## 5. 为什么第三卷不是“附加功能卷”

如果只把第三卷看成插件、MCP、remote 的集合，就会低估它在全书里的位置。第三卷真正做的是把 Claude Code 从一个本地 CLI 会话，推进成一套：

- 可以接入未知能力的系统；
- 可以组织多个行动者的系统；
- 可以跨机器、跨协议继续维持同一交互秩序的系统。

也就是说，第三卷不是给核心系统外接功能，而是在重新定义“核心系统的边界到底在哪里”。

所以第三卷最终留下的判断是：

> Claude Code 的扩展面与协作面，不是在核心系统之外长出的枝叶，而是在重新定义这套系统究竟能触达多远、能组织多少行动者、能承载多少运行形态。

## 与第四卷的关系

一旦第三卷把边界拉开，第四卷才有可能把这些入口、控制面、扩展面重新折叠回同一张系统地图。
